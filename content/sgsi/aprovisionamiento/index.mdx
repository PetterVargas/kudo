title: Aprovisionamiento

Bienvenido a la sección si deseas **aprovisionar** el SGSI en tu infraestructura o entorno corporativo.

En muchas ocasiones no contamos con un sitio donde estén las políticas y documentos de SGSI, por ello creamos un sitio fácil de implementar. El proyecto genera un sitio **completamente estático** (directorio `out/`) compatible con cualquier hosting de archivos estáticos.

<Steps>
<Step>

### Requisitos

- [X] Tener [Git](https://git-scm.com/install/) en tu equipo; para clonar la información.
- [X] Tener [Node.js 20+](https://nodejs.org/) y [pnpm](https://pnpm.io/installation), [npm](https://docs.npmjs.com/downloading-and-installing-node-js-and-npm) o yarn en tu equipo.
- [X] Un repositorio de código en [GitHub](https://github.com) o [GitLab](https://gitlab.com); para subir el código y automatizar el despliegue.
- [X] Un recurso de hosting estático — se recomienda [**Azure Static Web Apps**](https://azure.microsoft.com/es-es/products/app-service/static) o [**Cloudflare Pages**](https://pages.cloudflare.com/) (ambos tienen plan gratuito).
- [X] Opcional: un [Identity Provider (IdP)](https://en.wikipedia.org/wiki/Identity_provider) corporativo como Microsoft Azure Entra ID o Google Workspace para proteger el sitio y que sólo accedan colaboradores internos.

</Step>

<Step>

### Crear proyecto

Ejecuta el CLI de Kudo para generar la estructura base del proyecto:

```bash tab="pnpm"
pnpm create @divisioncero/kudo-app
```

```bash tab="npm"
npm create @divisioncero/kudo-app
```

```bash tab="yarn"
yarn create @divisioncero/kudo-app
```

El CLI te preguntará el nombre del proyecto y creará el directorio con toda la estructura de contenido lista para personalizar.

</Step>

<Step>

### Ejecución local

```bash tab="pnpm"
cd tu-proyecto
pnpm install
pnpm dev
```

```bash tab="npm"
cd tu-proyecto
npm install
npm run dev
```

```bash tab="yarn"
cd tu-proyecto
yarn install
yarn dev
```

Tu aplicación estará disponible en [http://localhost:3000](http://localhost:3000)

</Step>

<Step>

### Adaptación a tu organización

Antes de desplegar, personaliza el proyecto con los datos de tu organización o temas personales. Los cambios principales están en dos archivos:

#### 1. Identidad del sitio — `lib/shared.ts`

```ts
export const appName = 'Nombre de tu empresa'; // [!code highlight]

export const gitConfig = {
  user: 'tu-usuario-github',   // [!code highlight]
  repo: 'tu-repositorio',      // [!code highlight]
  branch: 'main',
};
```

#### 2. Contenido del SGSI — `content/sgsi/`

Edita los archivos `.md` dentro de `content/sgsi/politicas/` con las políticas reales de tu organización. Cada archivo corresponde a un documento del SGSI:

```
content/sgsi/politicas/
├── politica-de-seguridad-de-la-informacion.md  ← editar
├── politica-de-gestion-de-accesos.md           ← editar
├── politica-de-gestion-de-riesgos.md           ← editar
└── ...
```

#### 3. Branding — `app/layout.config.tsx`

Reemplaza el SVG del logo por el de tu organización y actualiza los enlaces de navegación según sea necesario.

</Step>

<Step>

### Probar los cambios

Antes de desplegar, verifica que el build estático se genera correctamente:

```bash tab="pnpm"
pnpm build
```

```bash tab="npm"
npm run build
```

```bash tab="yarn"
yarn build
```

Si el build es exitoso, se genera el directorio `out/` con el sitio estático completo. Puedes previsualizarlo localmente:

```bash tab="pnpm"
pnpm dlx serve out
```

```bash tab="npm"
npx serve out
```

Abre [http://localhost:3000](http://localhost:3000) y verifica que todas las páginas, políticas y navegación funcionen correctamente antes de continuar.

<Callout type="warn">
  Si el build falla, revisa que todos los archivos MDX tengan el frontmatter correcto (`title` y `description`) y que los enlaces internos en `meta.json` apunten a archivos existentes.
</Callout>

</Step>

<Step>

### Autenticación corporativa

Como el sitio es **estático**, la autenticación se configura a nivel del hosting, no en el código. Elige tu plataforma:

<Tabs items={["Azure Static Web Apps", "Cloudflare Pages"]}>
<Tab value="Azure Static Web Apps">

Azure SWA incluye autenticación nativa con Microsoft Entra ID. Crea el archivo `staticwebapp.config.json` en la raíz del proyecto:

```json title="staticwebapp.config.json"
{
  "auth": {
    "identityProviders": {
      "azureActiveDirectory": {
        "registration": {
          "openIdIssuer": "https://login.microsoftonline.com/<TENANT_ID>/v2.0",
          "clientIdSettingName": "AAD_CLIENT_ID",
          "clientSecretSettingName": "AAD_CLIENT_SECRET"
        }
      }
    }
  },
  "routes": [
    {
      "route": "/.auth/login/aad",
      "allowedRoles": ["anonymous"]
    },
    {
      "route": "/.auth/*",
      "allowedRoles": ["anonymous"]
    },
    {
      "route": "/*",
      "allowedRoles": ["authenticated"]
    }
  ],
  "responseOverrides": {
    "401": {
      "redirect": "/.auth/login/aad",
      "statusCode": 302
    }
  }
}
```

Luego configura las variables de entorno en el portal de Azure:
- `AAD_CLIENT_ID` → Client ID de tu App Registration en Entra ID
- `AAD_CLIENT_SECRET` → Client Secret generado en Entra ID

Para restringir el acceso **solo a usuarios de tu tenant**, en tu App Registration ve a **"Authentication"** y en **"Supported account types"** selecciona _"Accounts in this organizational directory only"_.

</Tab>
<Tab value="Cloudflare Pages">

Cloudflare Pages protege el sitio mediante **Cloudflare Access**, que se configura desde el dashboard sin tocar el código.

1. Ve a **Zero Trust → Access → Applications** en el [dashboard de Cloudflare](https://one.dash.cloudflare.com).
2. Crea una nueva aplicación de tipo **"Self-hosted"**.
3. En **Application domain** introduce el dominio de tu sitio (ej. `sgsi.tuempresa.com`).
4. En **Identity providers** conecta tu IdP corporativo:
   - **Microsoft Entra ID**: sigue la guía [Cloudflare + Azure AD](https://developers.cloudflare.com/cloudflare-one/identity/idp-integration/azuread/).
   - **Google Workspace**: sigue la guía [Cloudflare + Google](https://developers.cloudflare.com/cloudflare-one/identity/idp-integration/google/).
5. Crea una **Policy** que permita acceso solo a usuarios con email de tu dominio corporativo (ej. `*@tuempresa.com`).

<Callout>
  Cloudflare Access tiene plan gratuito para hasta 50 usuarios. Revisa los límites en [cloudflare.com/plans](https://www.cloudflare.com/plans/zero-trust/).
</Callout>

</Tab>
</Tabs>

</Step>

<Step>

### Configurar CI/CD

<Callout type="info">
  Si implementas directamente desde el portal de Azure o el dashboard de Cloudflare, la conexión con GitHub se realiza desde el propio CSP y el workflow de CI/CD se crea automáticamente en tu repositorio. Los pasos siguientes aplican solo si prefieres configurarlo manualmente.
</Callout>

Crea el workflow de GitHub Actions según tu plataforma de despliegue. El proyecto ya genera el sitio estático en `out/` con `pnpm build`.

<Tabs items={["Azure Static Web Apps", "Cloudflare Pages"]}>
<Tab value="Azure Static Web Apps">

Crea el archivo `.github/workflows/azure-swa.yml`:

```yaml title=".github/workflows/azure-swa.yml"
name: Deploy Azure Static Web App

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build_and_deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - name: Install dependencies
        run: pnpm install

      - name: Build
        run: pnpm build

      - name: Deploy
        uses: Azure/static-web-apps-deploy@v1
        with:
          azure_static_web_apps_api_token: ${{ secrets.AZURE_STATIC_WEB_APPS_API_TOKEN }}
          repo_token: ${{ secrets.GITHUB_TOKEN }}
          action: upload
          app_location: /
          output_location: out
          skip_app_build: true
```

Agrega el secret `AZURE_STATIC_WEB_APPS_API_TOKEN` en **GitHub → Settings → Secrets and variables → Actions**. El token lo obtienes en el portal de Azure al crear el recurso Static Web App.

</Tab>
<Tab value="Cloudflare Pages">

Crea el archivo `.github/workflows/cloudflare-pages.yml`:

```yaml title=".github/workflows/cloudflare-pages.yml"
name: Deploy Cloudflare Pages

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - name: Install dependencies
        run: pnpm install

      - name: Build
        run: pnpm build

      - name: Deploy to Cloudflare Pages
        uses: cloudflare/pages-action@v1
        with:
          apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          accountId: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
          projectName: mi-sgsi
          directory: out
          gitHubToken: ${{ secrets.GITHUB_TOKEN }}
```

Agrega en **GitHub → Settings → Secrets**:
- `CLOUDFLARE_API_TOKEN` → token con permiso `Cloudflare Pages: Edit` desde [dash.cloudflare.com/profile/api-tokens](https://dash.cloudflare.com/profile/api-tokens)
- `CLOUDFLARE_ACCOUNT_ID` → tu Account ID visible en el dashboard de Cloudflare

</Tab>
</Tabs>

</Step>

<Step>

### Desplegar en producción

Crea el recurso en tu plataforma elegida y conecta el repositorio:

<Tabs items={["Azure Static Web Apps", "Cloudflare Pages"]}>
<Tab value="Azure Static Web Apps">

1. En el [portal de Azure](https://portal.azure.com), busca **Static Web Apps** y haz clic en **Crear**.
2. Elige tu suscripción, grupo de recursos y dale un nombre al recurso.
3. En **Plan** selecciona **Free** o **Standard** (Standard requerido para auth con Entra ID).
4. En **Detalles de implementación** selecciona **GitHub** y autoriza el acceso.
5. Selecciona tu organización, repositorio y rama (`main`).
6. En **Build Details**:
   - **App location**: `/`
   - **Api location**: *(vacío)*
   - **Output location**: `out`
7. Haz clic en **Revisar y crear → Crear**.

Azure creará automáticamente el secret `AZURE_STATIC_WEB_APPS_API_TOKEN` en tu repositorio de GitHub y ejecutará el primer despliegue.

Una vez desplegado, configura un dominio personalizado en **Custom domains** dentro del recurso.

</Tab>
<Tab value="Cloudflare Pages">

1. En el [dashboard de Cloudflare](https://dash.cloudflare.com), ve a **Workers & Pages → Create → Pages**.
2. Haz clic en **Connect to Git** y autoriza el acceso a GitHub.
3. Selecciona tu repositorio y haz clic en **Begin setup**.
4. Configura el build:
   - **Project name**: `mi-sgsi` (debe coincidir con `projectName` en el workflow)
   - **Production branch**: `main`
   - **Build command**: *(vacío — el CI/CD ya construye antes de desplegar)*
   - **Build output directory**: `out`
5. Haz clic en **Save and Deploy**.

Para dominio personalizado, ve a **Custom domains** en tu proyecto de Pages y agrega tu dominio corporativo.

<Callout>
  Si usas el workflow de GitHub Actions con `cloudflare/pages-action`, en el paso 4 deja el **Build command** vacío porque el build ya ocurre en el runner de GitHub antes de hacer el upload a Cloudflare.
</Callout>

</Tab>
</Tabs>

Tras el primer despliegue exitoso, cada `push` a `main` actualizará el sitio automáticamente en pocos minutos.

</Step>
</Steps>
