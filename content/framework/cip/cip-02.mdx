title: CIP-02 - Ciberseguridad en Autenticación
description: Control de mecanismos de autenticación robusta
---

## Descripción del Control

Establece los requisitos de autenticación robusta para verificar la identidad de usuarios y servicios. Exige MFA obligatorio 
en todos los accesos administrativos, remotos y a sistemas críticos, con soporte para múltiples factores (app autenticadora, 
token hardware, biométrico). Implementa SSO corporativo mediante protocolos SAML 2.0/OAuth 2.0/OIDC, políticas de contraseñas 
fuertes (mínimo 14 caracteres, historial de 12), autenticación sin contraseña (FIDO2, Windows Hello) y monitoreo de credenciales 
comprometidas. Para servicios, exige managed identities y service principals con rotación de credenciales.

## Requisitos

### Requisito 1 - Autenticación Multifactor (MFA) Obligatorio

La organización implementa MFA obligatorio para todos los accesos administrativos, accesos remotos y accesos a sistemas y datos 
críticos o confidenciales. El MFA soporta múltiples factores de segunda verificación (aplicación autenticadora TOTP, token 
hardware FIDO2, biometría) y se aplica en el acceso inicial y ante operaciones sensibles (cambios de permisos, acceso a datos 
restringidos). La exención de MFA requiere aprobación formal del CISO y documentación de controles compensatorios.

### Requisito 2 - Single Sign-On (SSO) y Gestión de Contraseñas

La organización implementa SSO corporativo basado en protocolos estándar (SAML 2.0, OAuth 2.0, OIDC) para centralizar la autenticación 
a las aplicaciones corporativas y minimizar la proliferación de credenciales. Las políticas de contraseñas exigen mínimo 14 
caracteres, historial de 12 contraseñas anteriores y prohibición de contraseñas comprometidas verificadas contra bases de datos de 
credenciales comprometidas. Se promueve la autenticación sin contraseña (FIDO2, Windows Hello) como evolución hacia arquitecturas más seguras.

### Requisito 3 - Autenticación de Servicios y Cuentas No Humanas

Los servicios y aplicaciones que requieren autenticación entre sí utilizan managed identities o service principals con permisos 
mínimos, evitando el uso de credenciales estáticas de larga duración. Las API keys y tokens de servicio tienen fecha de expiración 
definida y son rotados automáticamente según la periodicidad establecida (mensual para API keys de producción). Las credenciales 
comprometidas detectadas (mediante monitoreo continuo o reportes de inteligencia de amenazas) son revocadas inmediatamente.
